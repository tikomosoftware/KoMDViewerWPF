# WinUI 3で詰まったMarkdownビューアを、Codexと一緒にWPFで作り直した話

## 1. 作りたかったもの

作りたかったのは、とてもシンプルなMarkdownビューアです。

目的は `.md` ファイルを開いて読むこと。  
高機能なエディタではなく、普段のメモや記事下書きを軽く確認できる、小さなWindowsデスクトップアプリを想定していました。

最終的にできたものは **KoMDViewer WPF** というWPFアプリです。

主な機能は以下です。

- Markdown / textファイルの表示
- 左側の履歴ペイン
- 履歴からの削除
- 履歴のファイル名順ソート
- ドラッグ&ドロップで開く
- ライト / ダークテーマ
- About画面への技術スタック表示
- アプリアイコンの差し替え

技術スタックは、かなり素朴です。

- .NET 9
- WPF
- Markdig
- Microsoft Edge WebView2
- System.Text.Json

## 2. 最初はWinUI 3で作ろうとしていた

もともとはWinUI 3で作ろうとしていました。

WinUI 3は見た目も新しく、WindowsアプリらしいUIを作れそうだったからです。ただ、今回の用途は「Markdownを開いて読むだけ」の小さなツールです。実際に進めてみると、Windows App SDKやランタイムまわり、配布まわりで考えることが多くなりました。

途中で、アプリの目的に対して構成が少し重くなっていると感じました。

そこでCodexに相談したところ、今回の用途ならWPFのほうが現実的だという方向になりました。

WPFなら、

- ファイルを開く
- ドラッグ&ドロップを受ける
- WebView2を貼る
- MarkdownをHTMLに変換して表示する

という流れが素直に書けます。

結果として、WinUI 3で頑張るよりも、WPFでシンプルに作り直す判断をしました。

## 3. Codexに任せた作業の進め方

今回の作業では、最初から細かい仕様書を用意したわけではありません。

最初に伝えたのは、だいたい次のような内容です。

> WinUI 3で作ろうとして諦めたアプリです。  
> `.md` ファイルを見るだけのシンプルなツールにしたいです。  
> WPFで作り直せますか。

Codexはまず既存プロジェクトを確認しました。

- `KoMDViewer.csproj`
- `App.xaml`
- `MainWindow.xaml`
- `MainWindow.xaml.cs`
- 既存のREADMEやビルドスクリプト

その結果、既存コードはまだWinUI 3 / Windows App SDKの構成で、CodeMirrorを使った編集機能やPDF出力なども含まれていました。

そこから、アプリの目的を「閲覧」に絞って、WPF版に置き換えていきました。

主な変更は以下です。

- `UseWinUI` をやめて `UseWPF` に変更
- Windows App SDK依存を削除
- WPFの `Window` / `Menu` / `StatusBar` に置き換え
- Markdown変換にMarkdigを使用
- 表示にWPF版WebView2を使用
- ファイル履歴を `%LocalAppData%` に保存

ここでよかったのは、Codexがいきなり大きく作り込むのではなく、まず「ビルドできる最小のWPFビューア」に寄せてくれたことです。

## 4. 作りながら追加した機能

最初にWPF版として起動できるところまで作ったあと、実際に触りながら追加機能を足していきました。

### 左側の履歴ペイン

最初はメニュー内に最近開いたファイルがあるだけでした。

ただ、Markdownビューアとして使うなら、左側に履歴が見えていて、選ぶと右側の表示が切り替わるほうが自然です。

そこで、画面を左右ペイン構成にしました。

- 左: 最近開いたファイル
- 右: Markdownプレビュー
- 中央: リサイズ用のSplitter

履歴を選ぶと右側のWebView2に表示するHTMLを差し替えます。

### 履歴の挙動修正

一度実装したあと、履歴をクリックするたびにそのファイルが履歴の一番上へ移動してしまいました。

これは「履歴から選んだ」操作まで「新しく開いた」扱いにしていたためです。

そこで、ファイルを開く処理に `addToRecentFiles` のようなフラグを持たせ、履歴選択時には並び順を変えないようにしました。

### ドラッグ&ドロップ

ドラッグ&ドロップも対応しました。

途中でWebView2の上にドロップ表示がうまく重ならず、左側だけに半透明の表示が出るような問題がありました。

これはWebView2が通常のWPFコントロールとは少し違う描画/ホストのされ方をするためです。

対策として、ドラッグ中は一時的にWebView2を隠し、ドロップ用オーバーレイを全面に出すようにしました。また、右側のWebView2ペイン上でもドラッグイベントを拾えるように、WebView2自身にもドラッグイベントを接続しました。

### 履歴の右クリック削除と並び替え

履歴から不要な項目を消せるように、右クリックメニューを追加しました。

削除するのは履歴からだけで、実ファイルは削除しません。

また、履歴の表示順も、

- 最近開いた順
- ファイル名順

を切り替えられるようにしました。

設定は `settings.json` に保存し、次回起動時にも維持されます。

## 5. アプリ名、アイコン、About画面

途中でアプリ名も見直しました。

一度は `KoMDReader WPF` という名前も考えましたが、最終的には **KoMDViewer WPF** にしました。

理由は単純で、このアプリは「読む」よりも「表示する」ツールだからです。Markdown Viewerという言い方のほうが、用途に合っています。

アイコンも作り直しました。

今回は生成AIで画像を作るのではなく、小さいサイズでも読みやすいようにSVGベースで作りました。

デザインは、

- 濃いネイビーの角丸タイル
- 白いドキュメント
- シアンの `MD`

というシンプルなものです。

WPFアプリのアイコンとして使うために、SVGソースを残しつつ、実際にはマルチサイズの `.ico` に変換しています。

About画面も、標準のMessageBoxではなく、専用の小さなWPFダイアログにしました。そこにアプリアイコンと技術スタックを表示しています。

## 6. 不要になったものを削除してREADMEも整理

WPF版として形になったあと、最後に不要ファイルを確認しました。

もともとのプロジェクトには、WinUI 3時代の名残として以下が残っていました。

- `editor/`
- `frontend/`
- Vite関連ファイル
- CodeMirror関連ファイル

今回のWPF版では使っていないため、削除しました。

READMEもかなり直しました。

古いREADMEには、まだ以下のような記述が残っていました。

- WinUI 3
- Windows App SDK Runtime
- CodeMirror編集モード
- PDF出力
- 動作不安定という注意書き

現在のWPF版には合わないため、READMEを作り直しました。

現在のREADMEでは、

- WPF版であること
- 必要ランタイムは .NET 9 Desktop Runtime と WebView2 Runtime であること
- 使い方
- 対応ファイル
- 技術スタック

だけをシンプルに書いています。

## 7. やってみて思ったこと

今回の作業でよかったのは、Codexに「完成形を一発で作らせる」のではなく、実際に動かしながら少しずつ直していけたことです。

たとえば、

- 起動できるか確認する
- 履歴ペインを追加する
- 履歴の挙動を直す
- ドラッグ表示のズレを直す
- アイコンを変える
- READMEを現状に合わせる
- 不要ファイルを削除する

という感じで、会話しながらアプリが育っていきました。

Codexはコードを書くところだけでなく、既存プロジェクトの棚卸しや、古いREADMEの矛盾を見つける作業にも向いていると感じました。

一方で、実際に画面を見て「この表示は変ではないか」と判断するのは、やはり人間側の役割です。今回も、ドラッグ時のオーバーレイのズレは、スクリーンショットを見せて初めて直せました。

最終的に、WinUI 3で詰まっていたMarkdownビューアを、WPFでシンプルに作り直すことができました。

小さなツールなら、最新UIフレームワークにこだわるより、目的に合う技術を選んだほうが気持ちよく進むこともある、という例になったと思います。
